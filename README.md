# Tentacula

Приложение Tentacula работает как терминирующий компонент. Когда клиенту нужно обратиться к удалённой системе, доступ к
которой напрямую невозможен или не рекомендуется, на разрыв между клиентом и сервером устанавливается Tentacula. По
HTTP/HTTPS-протоколу клиент передаёт данные необходимые для запроса (данные авторизации и атрибуты запроса), на заранее
опубликованную "Присоску". Присоска уже с терминирующего компонента обращается на конечный сервис (протокол между
Tentacula и сервером определяется в функции самой Присоски). После того как сервер даст ответ, терминирующий компонент
ретранслирует его клиенту в режиме стриминга.

Tentacula поддерживает 3 механизма взаимодейтсвия:

1. suсkers (присоски) - эндпоинт использующий POST-протокол, с заранее описанными требованиями и функционалом, который
   будет исполнен с сервера Tentacula. Предусмотрены встроенные эндпоинты для взаимодействия с DS;
2. composition (сочленение) - эндпоинт передачи клиентского запроса следующему за ним эндпоинту (в том чсиле и другим
   Tentacula);
   чтобы последняя в очереди Tentacula обратилась к серверу;
3. schedulers (планировщик) - механизм автозапуска заданий, которые необходимо исполнять с сервера Tentacula.
   Предусмотрен встроенный планиврощик исполнения задания к DS, когда планировщик ожидает задание в таблице БД.

## Авторизация на эндпоинтах

Эндпоинты требуют использовать один из способов аунтификации: анонимный (без проверки пользователя), по
сертификату (проверка по полю subject; требует использование проксирующим компонентам NGINX или подобным) и по LDAP в
сочетании авторизации на сервере DS и наличию членства в группе

## Установка

Пакет Tentacula подготовлен для установки в оффлайн режиме.
В папке `packages` указаны необходимые зависимости и готовые Conda-среды для Windows и Linux.
Настройка службы uvicorn происходит вне дистрибутива.
В файле `config.cfg` производится настройка приложения (в том числе формирование конфигурационного файла для NGINX с
рекомендуемыми параметрами).

### Установка зависимостей

Перед настойкой приложения требуется установка следующих зависимостей:

```
apt-get update && apt-get install -y \
krb5-user \
supervisor \
nginx
```

NGINX требуется если он будет использоваться как проксирующий элемент.

### Подготовка рабочей области

Рабочая область приложения структурируется следующим образом (на примере относительно пути `/app/tentacula`):

- Папка приложения `app` располагается в корне рабочей области;
- В корне рабочей области обязательно должны быть конфигурационный файл `config.cfg`;
- Сертификаты, папка с присосками и папка с планировщиком могут быть переопределены в любое место через `config.cfg`;

### Подготовка среды Conda

Для работы через среду конда, на UNIX-системах необходимо корректно развернуть Conda-Pack. Папка с распакованным
Conda-Pack'ом является рабочей средой и далее её нужно будет указать в параметрах конфигурирования.
Подготовка среды (на примере относительно пути `/app/tentacula`, таким образом рабочей средой будет
`/app/tentacula/conda_env`):

```
mkdir /app/tentacula/conda_env
tar -xzf tentacula.tar.gz -C /app/tentacula/conda_env
rm tentacula.tar.gz
```

### Настройка переменных окружения и запуск службы

Переменные окружения необходимо для работы (обратите внимание на область среды):

```
ENV PATH="/app/tentacula/conda_env/bin:$PATH"
ENV PYTHONDONTWRITEBYTECODE=1
ENV PYTHONUNBUFFERED=1
```

Строка запуска приложения как ASGI web-сервера (запускается из области приложения `/app/tentacula`):
`/app/tentacula/conda_env/bin/conda-unpack && uvicorn app.main:app --host 0.0.0.0 --port 8000`

## Настройка среды

Переменные среды читаются либо из окружения, либо из config.cfg
Приоритет чтения:

1. Переменные окружения;
2. Конфигурационный файл `config.cfg`;
3. Значение по умолчанию, если есть.

Переменные окружения генерируются по формуле: `TENTACULA__<РАЗДЕЛ>__<ПЕРЕМЕННАЯ>` (всё заглавными буквами)

Следующие атрибуты, если в начале будет написано `cat ` (с пробелом), значение атрибутов будет чистаться из файла, путь до которого указан после `cat `:

- `app` -> `DB_ASYNC_URL`;
- `app` -> `DB_SECRET_KEY`.
